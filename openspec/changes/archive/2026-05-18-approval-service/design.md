## 上下文

张江科学城综合党委当前使用 Excel 收集请示信息、手工匹配 Word 模板生成批复。系统涉及 8 类请示（成立、换届、增补、调整、更名、撤销、延期、架构调整），部分流程分多步（成立 3 步、换届 3 步、增补 2 步）。现有 14 个 Word 模板，变量包括 desttitle、name、title、ref_org、ref_person、meet_time 等。用户约 100 人，内网部署。

## 目标 / 非目标

**目标：**
- 申请者线上提交请示，替代 Excel 收集
- 系统自动匹配模板并生成批复文档（docx），保持模板格式
- 支持多步流程，每步独立请示，每步更新组织统计
- 四种角色（申请者/批复者/联络员/管理员）权限隔离
- 批复号自动递增
- 审批状态跟踪（待审批→已审批→已签批→已打印→已领取+退回修改）

**非目标：**
- 不做 AI 识别请示内容，纯表单采集
- 不做组织架构树形管理，统计字段手动维护
- 不做历史数据迁移，系统从零开始
- 不做移动端适配

## 决策

### 1. 技术栈选择

**决策**：Vue 3 + Node.js(Express) + SQLite

**理由**：
- Vue 3：轻量、学习成本低、适合中小项目
- Express：Node.js 生态成熟，快速开发
- SQLite：零配置、单文件、适合 100 用户内网场景

**替代方案**：
- React：生态更大但学习成本更高，本项目规模不需要
- NestJS：过度工程化，Express 足够
- PostgreSQL/MySQL：需要额外部署，SQLite 更简单

### 2. 文档生成方案

**决策**：使用 docxtemplater 库

**理由**：
- 直接操作 docx 模板，变量替换后保持原有格式（字体、行间距）
- 模板文件已包含完整格式设置，只需替换变量
- 支持条件判断和循环（处理多组织、多候选人场景）

**替代方案**：
- docx (npm)：需要代码中设置格式，无法继承模板样式
- python-docx：需要 Python 环境，增加部署复杂度

### 3. 认证方案

**决策**：JWT + Session 双重认证

**理由**：
- 内网部署，安全要求适中
- JWT 无状态，适合 API 调用
- 前端存储 token，每次请求携带

### 4. 模板匹配逻辑

**决策**：类别 + 进度值直接映射模板文件

**理由**：
- 映射关系固定（如"换届-1"→"换届1.docx"），无需复杂规则引擎
- 管理员可配置映射关系，存储在数据库
- 新增模板只需上传文件并配置映射

### 5. 数据库设计

**决策**：核心表设计

```
users: 用户表（id, username, password_hash, role, name）
organizations: 党组织表（id, name, parent_id, level）
user_org_auth: 用户组织授权（user_id, org_id）
templates: 模板表（id, category, step, filename, variables）
requests: 请示表（id, org_id, user_id, title, category, step, status, ...）
approvals: 批复表（id, request_id, batch_number, content, ...）
org_stats: 组织统计（id, request_id, ...统计字段）
```

**理由**：
- 请示表记录每次提交，包含类别和进度
- 批复表关联请示，记录批复文档和批复号
- 组织统计表记录每次请示对组织架构的影响

### 6. 批复号生成

**决策**：按年份 + 序号自动递增

**格式**：`沪张江科综委〔2026〕X号`，X 从 1 开始递增

**实现**：
- 数据库记录当前最大序号
- 每次审批通过时查询最大序号 +1
- 支持子区域编号（如浦软：`沪张江科综委浦软〔2026〕1号`）

## 风险 / 权衡

### 风险 1：模板变量不一致
**风险**：不同模板使用的变量名可能不一致（如 ref_person vs ref_person2）
**缓解**：管理员配置模板时统一定义变量名，系统校验变量完整性

### 风险 2：多步流程状态混乱
**风险**：用户可能跳过步骤或重复提交
**缓解**：系统校验流程顺序，如换届必须按 1→2→3 顺序提交

### 风险 3：组织统计数据不准确
**风险**：手动更新容易出错
**缓解**：提供统计汇总页面，管理员可对比校验

### 权衡：SQLite 并发限制
**权衡**：SQLite 写入并发有限，100 用户场景下基本够用
**缓解**：如未来用户增长，可迁移至 PostgreSQL
