## 上下文

当前系统采用前后端分离架构：
- **前端**：Vue 3 + Element Plus，使用 axios 调用后端 API
- **后端**：Express + TypeScript + better-sqlite3

现有的列表页面（请示、模板、组织、用户、批次号）一次性从后端加载所有数据，前端使用 computed 属性进行筛选和搜索。这种方式在数据量增大时会导致性能问题。

## 目标 / 非目标

**目标：**
- 为所有列表页面添加服务端分页功能
- 支持每页展示数量选择（10/20/50/100）
- 支持按条件筛选（状态、类型等）
- 支持关键词模糊查询
- 保持良好的用户体验和响应速度

**非目标：**
- 不改变现有的数据模型
- 不实现实时搜索（debounce 即可）
- 不添加高级筛选（如日期范围、多选筛选）

## 决策

### 1. 分页实现方式：服务端分页

**选择**：服务端分页（LIMIT/OFFSET）

**理由**：
- 数据库已有数据，SQL 查询原生支持 LIMIT/OFFSET
- 减少网络传输量，提升响应速度
- 前端只需处理当前页数据

**替代方案**：
- 前端分页：简单但传输所有数据，不适合大数据量
- 游标分页：适合无限滚动，但实现复杂，不适合传统分页组件

### 2. API 设计：统一响应格式

**选择**：所有列表接口返回统一的分页响应格式

```typescript
interface PaginatedResponse<T> {
  data: T[];
  pagination: {
    page: number;
    pageSize: number;
    total: number;
    totalPages: number;
  }
}
```

**理由**：
- 前端可以复用分页组件和逻辑
- 便于维护和扩展

### 3. 筛选和查询：服务端处理

**选择**：所有筛选和查询条件通过 URL 参数传递到后端

**参数设计**：
- `page`：页码（默认 1）
- `pageSize`：每页数量（默认 20）
- `keyword`：模糊查询关键词
- 其他筛选参数：如 `status`、`category` 等

**理由**：
- 减少前端内存占用
- 支持大数据量下的筛选
- 便于实现 URL 状态同步

### 4. 前端组件：使用 Element Plus 的 ElPagination

**选择**：使用 Element Plus 内置的分页组件

**理由**：
- 与现有 UI 框架一致
- 开箱即用，支持多种配置
- 样式统一

## 风险 / 权衡

### 风险 1：OFFSET 性能问题
**问题**：深分页（如第 1000 页）时 OFFSET 性能较差
**缓解**：
- 当前业务场景数据量有限（预计 < 10000 条）
- 可以添加最大页码限制
- 未来可考虑游标分页优化

### 风险 2：筛选条件复杂性
**问题**：不同列表页面的筛选条件不同
**缓解**：
- 设计灵活的参数传递机制
- 每个接口独立处理自己的筛选逻辑
- 保持筛选条件简单（单选、精确匹配）

### 风险 3：前端状态管理
**问题**：分页状态需要与 URL 同步
**缓解**：
- 使用 Vue Router 的 query 参数
- 监听路由变化自动刷新数据
- 保持状态简洁
