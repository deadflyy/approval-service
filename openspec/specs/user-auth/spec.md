## 新增需求

### 需求:用户登录
系统必须提供用户名密码登录功能，登录后返回 JWT token。

#### 场景:成功登录
- **当** 用户输入正确的用户名和密码
- **那么** 系统返回 JWT token，用户进入对应角色的首页

#### 场景:登录失败
- **当** 用户输入错误的用户名或密码
- **那么** 系统返回错误提示，不生成 token

### 需求:角色权限控制
系统必须支持四种角色：申请者(applicant)、批复者(approver)、联络员(liaison)、管理员(admin)，每种角色只能访问授权的功能。

#### 场景:申请者权限
- **当** 申请者登录系统
- **那么** 只能提交请示、查看自己提交的请示、上传附件

#### 场景:批复者权限
- **当** 批复者登录系统
- **那么** 可以查看所有请示、审批请示、退回修改

#### 场景:联络员权限
- **当** 联络员登录系统
- **那么** 只能查看分配给自己的请示、更新签批/打印/领取状态

#### 场景:管理员权限
- **当** 管理员登录系统
- **那么** 可以管理用户、模板、组织数据、批复号、查看所有请示

### 需求:用户组织授权
系统必须支持为申请者配置可提交请示的组织列表，一个用户可以代表多个组织提交请示。

#### 场景:查看可提交组织
- **当** 申请者进入请示提交页面
- **那么** 系统显示该用户被授权的组织列表供选择

#### 场景:未授权组织
- **当** 申请者尝试为未授权的组织提交请示
- **那么** 系统拒绝提交并提示无权限
